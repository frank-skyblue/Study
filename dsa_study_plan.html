The DP rule

Before writing a single line of code for any DP problem, write on paper: (1) what does dp[i] represent, (2) what is the recurrence, (3) what are the base cases. If you skip this step, you're memorizing solutions, not learning DP.

Core Pattern Skeletons

Memorize these. Write each one from memory once a day in Week 1. By Month 2 they should be automatic.

🌊
BFS — Level Order
Month 2 · Trees & Graphs
Use when: shortest path, level-by-level processing, assigning depths.
from collections import deque def bfs(root): if not root: return [] queue = deque([root]) levels = [] while queue: level = [] for _ in range(len(queue)): node = queue.popleft() level.append(node.val) if node.left: queue.append(node.left) if node.right: queue.append(node.right) levels.append(level) return levels
🔍
DFS — Post-Order
Month 2 · Trees & Graphs
Use when: aggregating values up a tree (children before parents).
def dfs(node): # base case — leaf if not node.children: return values[node] total = 0 for child in node.children: total += dfs(child) values[node] = total return total # call for every root for root in roots: dfs(root)
🔎
Binary Search
Month 3 · Core Patterns
Use when: sorted input, or "find minimum X such that condition holds".
def binary_search(nums, target): lo, hi = 0, len(nums) - 1 while lo <= hi: mid = (lo + hi) // 2 if nums[mid] == target: return mid elif nums[mid] < target: lo = mid + 1 else: hi = mid - 1 return -1
🔙
Backtracking
Month 3 · Core Patterns
Use when: generate all combinations, permutations, or subsets.
def backtrack(start, path, result): result.append(path[:]) # add copy for i in range(start, len(nums)): path.append(nums[i]) backtrack(i + 1, path, result) path.pop() # undo choice result = [] backtrack(0, [], result) return result
🪟
Sliding Window
Month 1 · Foundations
Use when: subarray/substring with constraint, variable or fixed window size.
def sliding_window(nums): left = 0 result = 0 window = {} # or a counter for right in range(len(nums)): # expand window window[nums[right]] = \ window.get(nums[right], 0) + 1 # shrink if invalid while not valid(window): window[nums[left]] -= 1 left += 1 result = max(result, right - left + 1) return result
📊
Prefix Sum
Month 1 · Foundations
Use when: repeated range sum queries, subarray sum = k.
def subarray_sum(nums, k): count = 0 prefix = 0 seen = {0: 1} # prefix_sum → count for num in nums: prefix += num # check if (prefix - k) seen before if prefix - k in seen: count += seen[prefix - k] seen[prefix] = seen.get(prefix, 0) + 1 return count
Principles for the Whole 6 Months

The people who don't improve grind 300 problems without reflection. These habits are what separate real progress from time spent.

01
Quality over quantity
One problem fully understood beats five problems half-understood. After every problem, ask: why does this solution work, not just does it work. Could you explain it to someone else?
02
Spaced repetition
Every Friday, redo 3 problems from earlier in the week without looking at your solution. Every month, redo 5 problems from the previous month. Memory fades — build it back deliberately.
03
Track your mistakes
Keep a running doc. Every time you get stuck or get something wrong, write down exactly what the gap was — not "I got it wrong" but "I didn't know how to handle the case where the tree has a single node." Patterns will emerge.
04
Before you code — structure first
Write down: (1) what data structures do I need, (2) what is the shape of the input, (3) what does the output look like. This habit alone prevents most implementation bugs.
05
The gate rule
Don't move to the next month until you pass the end-of-month check. The checks exist because skipping them means the next month builds on a weak foundation. A week extra in Month 2 is better than struggling through Month 3.
06
Don't skip the boring ones
Sliding window and two pointers feel easy but hard versions trip everyone up. Master them in Month 1. The problems that feel too simple are often the ones where people lose interview points on edge cases.
07
After solving — ask what slowed you down
Time yourself. After the timer: what slowed me down? Which pattern was I uncertain about? Could I write that pattern from scratch tomorrow? Three questions, every session.
